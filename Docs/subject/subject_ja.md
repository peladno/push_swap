# Push_swap(日本語版サブジェクト)

> 原文: `en.subject.pdf` / Version 1.1
> *Because Swap_push doesn't feel as natural*

**概要:** 限られた命令セットを使い、できるだけ少ない操作回数でスタック上のデータをソートする課題。
さまざまな種類のアルゴリズムを操作し、最適化されたデータソートのために最も適した解(複数の中から)を選ぶ。

> ⚠️ **これは「ちょうど 2 人」の学習者で完成させるグループ課題**(ペア課題)。

---

## I. Foreword(まえがき)

Big-O 記法(計算量の増え方を表す記法)の話。入力が大きくなると、巧妙な O(n log n) 戦略は不格好な O(n²) を必ず上回る。`pb; pa; pb; pa; ...` を 10,000 回並べて checker に嫌われる人にならないように。**Sort smart, not slow.**

## IV. Introduction(導入)

ゴール: `push_swap` という C プログラムを書く。引数で受け取った整数を**ソートする最小の操作プログラム**(Push swap 言語の操作で構成)を計算し、標準出力に表示する。

## V. Objectives(目的)

このプロジェクトの目的は、**アルゴリズムの計算量を具体的に体感すること**。

- 数をソートするのは簡単。しかし 2 つのスタックと限られた命令だけで**速く**ソートするのは別の話。
- 完全にランダムなリストのソートと、ほぼソート済みのリストのソートは**まったく別物**。
- アルゴリズムの選択次第で「あっという間の勝利」と「終わりなき操作の羅列」に分かれる。

### VI.2 ルール

- `a`, `b` という 2 つのスタックを持つ。
- 開始時:
  - スタック `a` には **重複なし** の正負ランダムな数が入っている。
  - スタック `b` は空。
- ゴール: スタック `a` の数を **昇順** にソートする。使える操作:

| 操作 | 意味 |
|---|---|
| `sa` (swap a) | `a` の先頭 2 要素を入れ替える。要素が 1 個以下なら何もしない。 |
| `sb` (swap b) | `b` の先頭 2 要素を入れ替える。要素が 1 個以下なら何もしない。 |
| `ss` | `sa` と `sb` を同時に実行。 |
| `pa` (push a) | `b` の先頭要素を取り出し `a` の先頭に積む。`b` が空なら何もしない。 |
| `pb` (push b) | `a` の先頭要素を取り出し `b` の先頭に積む。`a` が空なら何もしない。 |
| `ra` (rotate a) | `a` の全要素を 1 つ上にシフト。先頭が末尾になる。 |
| `rb` (rotate b) | `b` の全要素を 1 つ上にシフト。先頭が末尾になる。 |
| `rr` | `ra` と `rb` を同時に実行。 |
| `rra` (reverse rotate a) | `a` の全要素を 1 つ下にシフト。末尾が先頭になる。 |
| `rrb` (reverse rotate b) | `b` の全要素を 1 つ下にシフト。末尾が先頭になる。 |
| `rrr` | `rra` と `rrb` を同時に実行。 |

### VI.3 アルゴリズム要件

時間・空間の計算量を厳密に理解させるため、**互いに異なる 4 つのソート戦略を実装**し、`push_swap` プログラムに統合する。プログラムは**実行時に入力構成に応じて戦略を選択**できなければならない。

#### VI.3.1 計算量モデルと操作の制約

すべての戦略は C で実装し、ソートを行う Push_swap 操作のシーケンスを生成する。つまり:

- C アルゴリズムが入力を分析し、適切な操作シーケンス `sa, sb, ss, pa, pb, ra, rb, rr, rra, rrb, rrr` を生成する。
- 戦略の出力は、スタックをソートするこれら操作のシーケンス。
- 計算量クラスを述べるときは、**生成された Push_swap 操作の数で測ったコスト**を反映すること(古典的な配列ベースアルゴリズムの理論計算量ではない)。

#### VI.3.2 Disorder 指標(必須)

このサブジェクトにおいて **disorder** は **0 と 1 の間の数値** で、初期スタック `a` がソート済みからどれだけ離れているかを表す。

- 既に正しい順序 → disorder = `0`
- 最悪の順序 → disorder = `1`
- その中間 → 部分的にソート済みだが乱れている

計算方法: スタック内のすべての数のペアを見る。**大きい数が小さい数より前に現れる**たびに、そのペアを 1 つの *mistake* としてカウント。mistake が多いほど disorder は 1 に近づく。

```text
function compute_disorder(stack a):
    mistakes = 0
    total_pairs = 0
    for i from 0 to size(a)-1:
        for j from i+1 to size(a)-1:
            total_pairs += 1
            if a[i] > a[j]:
                mistakes += 1
    return mistakes / total_pairs
```

> 補足: これは「転倒数(inversions)/ 全ペア数 C(n,2)」。disorder は **どの移動も行う前に** 測定すること。

#### VI.3.3 必須の戦略

1. **Simple アルゴリズム (O(n²))**: O(n²) クラスのベースラインを最低 1 つ実装。
   - 挿入ソートの応用 / 選択ソートの応用 / バブルソートの応用 / 単純な min・max 抽出法 など

2. **Medium アルゴリズム (O(n√n))**: O(n√n) クラスを最低 1 つ実装。
   - チャンクソート(√n 個のチャンクに分割)/ ブロック分割法 / √n バケットのバケットソート応用 / 範囲ベースのソート戦略 など

3. **Complex アルゴリズム (O(n log n))**: O(n log n) クラスを最低 1 つ実装。
   - 基数ソートの応用(LSD または MSD)/ 2 スタックを使うマージソート応用 / スタック分割のクイックソート応用 / ヒープソート応用 / BIT(Binary Indexed Tree)系 など

4. **Custom adaptive アルゴリズム(学習者の設計)**: 測定した **disorder** に応じて内部手法を選択する **適応的** 戦略。特定の名前付きアルゴリズムに縛られない(内部技法は完全に自由)。ただし以下の **計算量目標**(Push_swap 操作モデルにおいて)を守ること:

   | レジーム | 条件 | 要求される時間計算量 |
   |---|---|---|
   | Low disorder | disorder < 0.2 | O(n²) |
   | Medium disorder | 0.2 ≤ disorder < 0.5 | O(n√n) |
   | High disorder | disorder ≥ 0.5 | O(n log n) |

   リポジトリ(例: `README.md`)に、**しきい値の根拠**・**各レジームで使う内部技法**・**Push_swap モデルでの時間と空間の計算量議論(上界)** を文書化すること。

### VI.4 例

ランダムな整数リストをソートする例(両スタックは右から伸びると仮定)。`sa`, `pb pb pb`, `ra rb`(= `rr`), `rra rrb`(= `rrr`), `sa`, `pa pa pa` を順に実行すると、**12 操作**でスタック `a` がソートされる。「もっと良くできる?」

### VI.5 `push_swap` プログラム

| 項目 | 内容 |
|---|---|
| Program Name | `push_swap` |
| Files to Submit | `Makefile`, `*.h`, `*.c` |
| Makefile | `NAME`, `all`, `clean`, `fclean`, `re` |
| Arguments | stack a: 整数のリスト |
| External Function | `read`, `write`, `malloc`, `free`, `exit` / `ft_printf` または**自作の**同等品 |
| Libft authorized | Yes |
| Description | Sort stacks |

ルール:

- Makefile はソースをコンパイルし、**再リンクしない**こと。
- **グローバル変数は禁止**。
- `push_swap` は次を引数に取る:
  - 整数リストとしてフォーマットされたスタック `a`(**最初の引数がスタックのトップ**)。
  - オプションの **戦略セレクタ**:
    - `--simple` : O(n²) アルゴリズムを強制。
    - `--medium` : O(n√n) アルゴリズムを強制。
    - `--complex` : O(n log n) アルゴリズムを強制。
    - `--adaptive` : disorder に基づく適応アルゴリズムを使う。**セレクタなしのデフォルト動作**。
- スタック `a` をソートする **可能な限り最小の Push_swap 操作リスト** を表示(最小の数がトップ)。
- 操作は `\n` のみで区切る(それ以外なし)。
- 各アルゴリズムで主張する計算量クラスはこのモデルで有効であること。
- 戦略選択は**すべての有効な入力**で動作すること。どのセレクタフラグも入力サイズや disorder に関係なく動作すること。
- パラメータが指定されない場合、**何も表示せずプロンプトに戻る**。
- エラー時は標準エラーに `"Error"` + `\n`。エラー例: 整数でない引数、範囲外の整数、重複値。
- バイナリは **4 つの戦略すべて**(Simple O(n²)、Medium O(n√n)、Complex O(n log n)、Adaptive)を埋め込むこと。選択された戦略名と計算量クラスは `--bench` モードで参照できること。
- オプションの **ベンチマークモード (`--bench`)** はソート後に表示する:
  - 計算された **disorder**(% 表記、小数 2 桁)
  - 使われた戦略名とその理論計算量クラス
  - 操作の総数
  - 各操作タイプのカウント(`sa, sb, ss, pa, pb, ra, rb, rr, rra, rrb, rrr`)
  - bench 出力は **stderr** へ送り、フラグがあるときのみ表示する。

## VIII. Bonus part(ボーナス)

シンプルなプロジェクトなので追加機能の余地は限られる。そこで **自作の checker** を作る。

> checker プログラムにより、push_swap が生成した操作リストが実際にスタックを正しくソートするか確認できる。

> ⚠️ **ボーナスは必須部分が「完璧」な場合のみ評価される。** 完璧とは必須部分が完全に完成しエラーなく動作すること。本プロジェクトでは**すべてのベンチマークを例外なく検証**することを意味する。必須要件をすべて通していなければボーナスは一切評価されない。

### VIII.1 `checker` プログラム

| 項目 | 内容 |
|---|---|
| Program Name | `checker` |
| Files to Submit | `*.h`, `*.c` |
| Makefile | `bonus` |
| Arguments | stack a: 整数のリスト |
| External Function | `read`, `write`, `malloc`, `free`, `exit` / `ft_printf` または自作の同等品 |
| Libft authorized | Yes |
| Description | Execute the sorting operations |

- `checker` は整数リストとしてフォーマットされたスタック `a` を引数に取る(最初の引数がスタックのトップ。順序に注意)。引数なしなら停止し何も表示しない。
- その後、標準入力から操作を読む(各命令の後に `\n`)。**すべての命令を読み終えてから**、引数で受け取ったスタックに対して実行する。
- 実行後、スタック `a` が実際にソート済みで、かつスタック `b` が空なら、標準出力に `"OK"` + `\n`。
- それ以外のすべての場合、標準出力に `"KO"` + `\n`。
- エラー時は **標準エラー** に `"Error"` + `\n`。エラー例: 整数でない引数、int より大きい引数、重複、存在しない/不正にフォーマットされた命令。

```text
$> ./checker 3 2 1 0
rra
pb
sa
rra
pa
OK
$> ./checker 3 2 1 0
sa
rra
pb
KO
$> ./checker 3 2 one 0
Error
$> ./checker "" 1
Error
$>
```

> ⚠️ 提供バイナリと**まったく同じ挙動を再現する必要はない**。エラー管理は必須だが、引数のパース方法は自由。

### このサブジェクトが「古典版」と違う点(注意)

一般に出回っている push_swap 解説の多くは「1 アルゴリズム(クイック/基数)で全部ソート」だが、**この版は別物**:

- **4 つの異なる戦略**を内蔵し、`--simple/--medium/--complex/--adaptive` で切り替え必須。
- 各戦略の計算量クラスは「**Push_swap 操作数**」で測る(配列ソートの理論計算量ではない)。
- **disorder 指標**(転倒数 / 全ペア数)の実装が必須で、adaptive はこれでレジームを分岐。
- `--bench`、性能ベンチ(100 個 <2000、500 個 <12000 で合格)、グループ課題という制約。

### 検討すべき設計判断(ペアで話し合うポイント)

- スタックのデータ構造(配列 / 連結リスト)— どちらにする? 根拠は?
- 各戦略にどの具体アルゴリズムを採用するか(O(n²) / O(n√n) / O(n log n))。
- adaptive のしきい値(0.2, 0.5)と各レジームの内部技法 — README で正当化が必要。
- disorder の O(n²) 計算は 500 要素で問題ないか(全ペア = 約 12 万、許容範囲か)。
- 引数パース・エラー判定(INT 範囲外・重複・非整数)とセレクタフラグの位置の扱い。
