### What I did

Finished the benchmark counting layer (Phase 8, stage 2) and verified it end-to-end.

**Benchmark stage 1 — `bench_print` finished**

- Implemented `bench_print` (to stderr, gated on `bench->enabled`): disorder as `XX.XX%`, `total_ops`, and the 11 per-op counts split across two lines.
- **Printing a double without `%f`.** Our `ft_printf_err` supports `c/s/d/i/u/x/%` but not floats, and push_swap can't use `printf`. So I print the disorder by decomposing it into integers: `total = disorder * 10000`, `int_part = total / 100`, `frac_part = total % 100`.
- Two subtleties caught at review:
  1. **Leading zero of the fraction.** `0.6 → frac 0` printed as `"60.0%"` instead of `"60.00%"`. `ft_printf_err` has no `%02d`, so I emit a manual `'0'` when `frac < 10`. My first attempt forgot the `else`, so when `frac < 10` it printed the line *twice* — a classic if-without-else trap that only breaks on one branch.
  2. **Truncation vs rounding.** `(int)x` can turn a value that should be `2900` into `2899`, because a double stores e.g. `0.29` as `0.28999…`. Switched to `(int)(x + 0.5)` to round.
- `total_ops` sums `count[]` in a loop over `OP_COUNT` — the array layout (decision D1) pays off here.

**Benchmark stage 2 — op counting threaded through `t_stack`**

- **Design ① — the strategy string.** Output is `Adaptive / O(n√n)` etc.: the name stays the requested strategy, the complexity reflects the *actually chosen* algorithm. A single config enum can't express "Adaptive chose Medium," so the composed string is produced at the dispatcher (Phase 7) and `t_bench` just stores a `char *` (points to a string literal — no malloc). `bench_print` only prints it; the enum→string mapping never lives in the printer.
- **Design ② — keep `t_config` and `t_bench` separate.** Different consumers/lifetimes: `t_config` is read once by the dispatcher, `t_bench` is threaded through all 11 wrappers. `benched` (config) and `enabled` (bench) are the same boolean by *deliberate* copy — the bridge that lets the op layer stay ignorant of parsing.
- **Method B over A.** Instead of adding a `t_bench *` parameter to all 11 wrappers + every tier/helper (method A), I embedded `t_bench *bench` in `t_stack`. The wrappers already take `t_stack *`, so they reach the counter via `stack->bench` with **no signature changes** — which leaves Javier's medium/simple tiers untouched and avoids merge churn. Chose this explicitly because A's ripple would edit his files.
- **Both stacks must carry the pointer.** `sb`/`rb`/`rrb` only receive `stack_b`, so they read `stack_b->bench`; the two-stack wrappers read `stack_a->bench`. For every count to land in one `t_bench`, main wires `stack_a.bench` and `stack_b.bench` to the same `&bench`. (Spotted this myself before wiring.)
- **Counting rule: count only on emit.** Each wrapper increments its slot inside the same branch that prints the op (the `OP_DONE` block, or after the early-return guard for `ss`/`rr`/`rrr`), so counts always match the stdout op stream. Guarded with `if (stack->bench)` (decision D3-b) so non-benched runs (NULL pointer) are safe.

**Bugs caught during review**

- `sa`/`sb`: the increment sat *outside* the `OP_DONE` block, so a no-op swap (size < 2) would over-count. Moved it inside.
- `rb`: counted `OP_SB` instead of `OP_RB` (copy-paste from swap). `total_ops` stays correct, so the per-op breakdown was silently wrong — easy to miss.
- `rr`/`rrr`: 3-tab indentation (norm).
- Header didn't compile: `t_stack` referenced `t_bench` before its typedef. **A typedef name must be declared before it's used.** Reordered to `e_op_id → s_bench → s_stack` (the alternative was a `struct s_bench *` forward reference). norminette can't catch this — only a real build does.

**Subject re-check (a hypothesis that turned out wrong)**

- I suspected `--bench` should *suppress* the operation output in the wrappers. The subject says the opposite: the operation stream **stays on stdout** even in bench mode; "hide operations" is done by the shell (`2> bench.txt | ./checker`), exploiting stdout/stderr separation. Suppressing in the wrapper would break the documented `| ./checker` usage. So no wrapper change — `enabled` only gates `bench_print`. Reinforced the Unix model: stdout = data for the next command, stderr = human-facing diagnostics.
