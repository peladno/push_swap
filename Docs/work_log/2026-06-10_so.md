**Phase 6 — Radix Sort LSD (binary) core implemented**

- Found `Docs/implementation_plans/complex-radix-lsd.md` already existed (written in the pre-kickoff prep). Updated it: dropped the stale `t_bench *bench` parameter (op counting is a Phase 8 concern per operations.md — keeps the sort unaware of benchmarking) and renamed `radix_sort_lsd` → `radix_sort` to match `insertion_sort` / `chunk_sort`. Committed to main, then branched `so/phase6-radix`.
- Implemented `src/algorithms/complex/radix_sort.c` (3 functions):
  - `count_bit(size)`: `ceil(log2(n))` via `while ((1 << bit) < size) bit++`.
  - `process_bit(a, b, bit)`: one LSD pass. Snapshots `a->size` before the loop (critical — `pb` shrinks `a->size` mid-loop), distributes (0-bit → `pb`, 1-bit → `ra`), then merges `b` back with `pa`.
  - `radix_sort(a, b)`: early-return on size < 2, loops over `max_bit` passes.
- Two bugs caught at review:
  1. **Operator precedence**: `& 1 == 0` parses as `& (1 == 0)` = `& 0` = always 0, so `pb` never fired. C ranks `==` above `&`. Fixed with `((... & 1) == 0)`.
  2. **main.c error path** had `stack_init(&stack_b)` where `stack_free(&stack_b)` was meant (harmless since b is empty there, but asymmetric with the success path). Fixed.
- Verified on `3 1 2` → `1 2 3`, and traced the two bit-passes by hand to confirm the LSD mechanism. Then swept `5 8 -4 -3 1` / `1 2 3 4 5` / `5 4 3 2 1` / `2 1` / `42` — all sort ascending. **Non-adaptive confirmed**: all n=5 inputs take exactly 25 ops regardless of order (sorted, reverse, random) — the defendable property for the O(n log n) complexity argument. Negative numbers handled transparently because Radix only sees the compressed index. valgrind clean.
