**M4 design discussions (no implementation yet)**

Topics worked through before starting M4 code:

1. **Quoted string support**:
   - Subject's 7 examples in VI.5.1 all use multi-arg form (direct args, unquoted `$ARG`, unquoted `$(cat args.txt)`); no quoted form appears in the subject itself.
   - 42 peer-review culture sometimes uses `./push_swap "3 1 2"` style — so quoted form should still be supported defensively.
   - Decided unified approach: each `argv[i]` is independently fed to `ft_split(arg, ' ')`. The argv boundary carries no semantic meaning.
   - This handles all of: multi-arg, single quoted string, multiple quoted strings (e.g. `"1 2 4 5" " 9 0 3"`), mixed, and unquoted shell expansion.
   - Confirmed via examples that `$(cat args.txt)` and `$ARG` (both unquoted) are already word-split by the shell into separate argv entries — they don't trigger quoted-string handling.

2. **INT range validation strategy** (for the `safe_atoi` helper):
   - Compared three accumulator types: `size_t`, `long`, `long long`.
   - `size_t`: unsigned, requires separate sign handling and risks undefined behavior near `INT_MIN`. Reject.
   - `long`: width is implementation-dependent (LP64 = 64-bit on Linux/macOS, LLP64 = 32-bit on Windows 64-bit). Works on the campus environment but not portable.
   - `long long`: C99 guarantees ≥ 64 bits everywhere. **Chosen** for portability and clarity.
   - Algorithm: parse sign, accumulate in `long long`, early-reject when digit count > 11, validate trailing characters, then check `INT_MIN ≤ value ≤ INT_MAX`.

3. **parse_args memory management**:
   - Concerned about how to "concatenate" argv tokens. Resolved by removing concatenation entirely — process each `argv[i]` with `ft_split`, validate and push tokens immediately, then `free_split` the result.
   - Established the need for a small `free_split(char **)` helper to release `ft_split` output (the array + each string).
   - Decided duplicate detection walks the stack directly (two-pointer linked-list traversal), avoiding an extra temp int array. O(n²) C-side, 0 push_swap operations.
   - Error-path cleanup must handle the in-flight `tokens` array and the partially built stack (the latter via a static `clear_stack` helper inside `parse_args.c`, since `stack_pop_top` is intentionally absent).

**M4 設計議論(実装はまだ着手していない)**

M4 コード着手前に詰めたトピック:

1. **クォート文字列対応**:
   - subject VI.5.1 の 7 例はすべて複数引数形式(直接、クォートなし `$ARG`、クォートなし `$(cat args.txt)`)。subject 本体にクォート形式は登場しない。
   - 42 のピアレビュー現場では `./push_swap "3 1 2"` 形式でテストされることがあるため、防衛的に対応する方針。
   - 統一アプローチに決定:各 `argv[i]` を独立に `ft_split(arg, ' ')` で分解。**argv の境目は意味を持たない**。
   - これで複数引数 / 単一クォート / 複数クォート(例:`"1 2 4 5" " 9 0 3"`)/ 混在 / クォートなし shell 展開 すべてカバーできる。
   - `$(cat args.txt)` と `$ARG`(いずれもクォートなし)は shell が単語分割するので、最初から argv に複数要素として渡る点を確認。

2. **INT 範囲チェック(`safe_atoi` ヘルパー)の型選択**:
   - 3 つの蓄積型を比較:`size_t`、`long`、`long long`。
   - `size_t`:unsigned で符号別管理、`INT_MIN` 周りに未定義動作の罠 → 不採用。
   - `long`:実装依存(LP64 で 64bit、LLP64 で 32bit)。キャンパス環境では動くが移植性なし。
   - `long long`:C99 が 64bit 以上を保証。**採用**。
   - アルゴリズム:符号判定 → `long long` で蓄積 → 11 桁超で early reject → 末尾文字 check → `INT_MIN ≤ value ≤ INT_MAX` 確認。

3. **parse_args のメモリ管理**:
   - argv の「連結」をどうするか不安があった → 連結を一切やめる方針に。各 `argv[i]` を `ft_split` → 即 validate + push → `free_split` で解放。
   - `ft_split` 結果(配列 + 各文字列)を解放する `free_split(char **)` ヘルパーが必要。
   - 重複検出はスタック走査で実施(2 重ループの linked-list traversal)。一時 int 配列を作らない。C 側 O(n²)、push_swap 操作生成は 0。
   - エラー経路の後片付けは、処理中の `tokens` 配列と途中まで構築したスタックを扱う必要(後者は `parse_args.c` 内の static `clear_stack` ヘルパーで対応 ——`stack_pop_top` を意図的に削除した分の代替)。
