### What I did

**Phase 1 — M4 refactor: introduce `t_status` enum**

- Added a `t_status` enum (`STATUS_OK = 0`, `STATUS_ERROR = 1`) to `include/push_swap.h` to give success/failure return values a self-documenting type.
- Refactored four functions to return `t_status`: `safe_atoi`, `validate_token`, `push_arg_tokens` (static), `parse_args`.
- Kept `int 0/1` for boolean predicates (`has_duplicates`, `check_strategy`) — `t_status` is reserved for success/failure semantics, not yes/no questions.
- Discussion on coding style: chose `!= STATUS_OK` over `== STATUS_ERROR` for the if-condition idiom. Rationale: when more statuses are added later (e.g., `STATUS_OVERFLOW`), the `!= STATUS_OK` form keeps catching all non-success cases without modification.

**Extended `t_status` scope and cleaned up adjacent functions**

- `stack_push_bottom`: was `int 0/1`, refactored to `t_status`. **Important side effect**: revealed a latent bug — `push_arg_tokens` was ignoring its return value, so a `malloc` failure inside `stack_push_bottom` would silently skip the node and proceed. The refactor forced an explicit `!= STATUS_OK` check, closing the hole.
- `coord_compress`: was `int 0/1` with a NULL guard whose return was never checked by the caller. Changed to `void` — the function genuinely cannot fail (no malloc inside), and the NULL guard was defensive against an impossible state. Type-honest is better than dishonest type.
- Merged `src/parser/safe_atoi.c` into `src/parser/validate_token.c`, making `safe_atoi` a `static` function. Rationale: `safe_atoi` was only ever called by `validate_token`, so encapsulation wins (smaller public API, narrower header).
- Removed `safe_atoi.c` from `Makefile` SRCS and from `push_swap.h` prototypes.

**Extracted `process_one_token` to stay under the 25-line Norm limit**

- Adding the `stack_push_bottom` return check would have pushed `push_arg_tokens` to 27 lines.
- Compared three refactor options: extract a per-token helper / extract a cleanup helper / use a status-flag loop.
- Chose option A: `static t_status process_one_token(char *token, t_stack *stack_a)` — combines `validate_token` + `stack_push_bottom` for one token. Clean single responsibility, easy to explain at peer evaluation.
- `push_arg_tokens` ended at 21 lines, comfortably within Norm.

**Critical bug caught at review: incorrect `free_split` ownership**

- First draft of `process_one_token` called `free_split(&token)` on failure paths — taking the address of a local `char *` and passing it as if it were a malloc'd `char **` array.
- This would have caused undefined behavior (walking adjacent stack memory until a NULL was hit, then `free`'ing a stack address) in real error inputs.
- Root cause: confusion about ownership. `tokens` (the array) is owned by `push_arg_tokens`, not by `process_one_token`. Workers should not free what they don't own.
- Fixed by: removing all `free_split` calls from `process_one_token`, and adding the cleanup to `push_arg_tokens` on the per-token failure path.
- **Takeaway** (added to mental notes): ownership in C is not expressible in the type system — `char **` doesn't tell you whether you're allowed to free it. Default rule: "if you didn't malloc it, don't free it" unless explicitly transferred.

**Phase 1 — M4 リファクタ:`t_status` enum 導入**

- 4 関数の戻り値を `t_status` にリファクタ:`safe_atoi`、`validate_token`、`push_arg_tokens`(static)、`parse_args`。
- bool 系 predicate(`has_duplicates`、`check_strategy`)は `int 0/1` のまま維持 ——`t_status` は success/failure 専用、yes/no 質問には使わない方針を確立。
- コーディングスタイル議論:if 条件式の慣用句として `== STATUS_ERROR` ではなく `!= STATUS_OK` を採用。理由:将来 `STATUS_OVERFLOW` 等が増えても `!= STATUS_OK` 形なら無修正で「成功以外を全部捕捉」できる。

**`t_status` のスコープ拡張と周辺関数の整理**

- `stack_push_bottom`:`int 0/1` から `t_status` へ。**重要な副次効果**:潜在バグを露呈した —— `push_arg_tokens` が戻り値を捨てていたため、内部の `malloc` 失敗が静かに無視されていた。リファクタで `!= STATUS_OK` チェックを強制でき、穴を塞いだ。
- `coord_compress`:`int 0/1` で NULL ガード(戻り値は呼び出し側で check されず)。`void` に変更 —— 関数内に malloc がなく真に失敗しえないため、NULL ガードは「起こりえない状態」への防御だった。型が嘘をついていない方が良い。
- `src/parser/safe_atoi.c` を `src/parser/validate_token.c` にマージし、`safe_atoi` を `static` 化。理由:`safe_atoi` は `validate_token` からしか呼ばれず、カプセル化したほうが public API が狭くなる。
- `Makefile` SRCS から `safe_atoi.c` を削除、`push_swap.h` のプロトタイプも削除。

**`process_one_token` を抽出して 25 行 Norm 制約に収める**

- `push_arg_tokens` は 21 行に収まり、Norm に余裕。

**レビューで重大バグ発見:`free_split` の ownership 誤り**

- `process_one_token` 初版が失敗パスで `free_split(&token)` を呼んでいた —— ローカルな `char *` のアドレスを `free_split` に渡し、malloc 配列のように扱おうとしていた。
- これは未定義動作:スタック上の隣接メモリを NULL に当たるまで walking し、最後にスタックアドレスを `free` —— 実際のエラー入力で発火していたはず。
- 根本原因:ownership の混乱。`tokens` 配列の owner は `push_arg_tokens` であって `process_one_token` ではない。worker は自分が所有しないものを free すべきでない。
- 修正:`process_one_token` から `free_split` の呼び出しを全削除、cleanup は `push_arg_tokens` 側で per-token 失敗パスに追加。
- **学び**(メモに記録):C の ownership は型では表現できない —— `char **` を受け取ったからといって free する権利があるとは限らない。デフォルトルール「malloc した本人が free する、他人は触らない(明示譲渡がなければ)」。
