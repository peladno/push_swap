**Caught a design problem: `ss` / `rr` / `rrr` were silently mutating one stack**

Looking at the final test (`ss` with b size 1), the output showed: **no "ss\n" printed (as expected), BUT stack a's top two elements were swapped**. The non-atomic implementation called `swap(a)` (which succeeded) and `swap(b)` (which returned NOT_DONE), then suppressed the print — but the mutation on `a` persisted.

**Why this matters**: `checker_42` (the subject-mandated verifier) replays printed op lines against the initial input and compares against the program's final state. A silent mutation that no op line explains → KO. This is a guaranteed verification failure whenever a dual-stack wrapper sees one stack below size 2.

**Fixed by making `ss` / `rr` / `rrr` atomic**: precondition check on both stacks BEFORE any mutation:

```c
t_operation  ss(t_stack *a, t_stack *b)
{
    if (a->size < 2 || b->size < 2)
        return (OP_NOT_DONE);
    swap(a);
    swap(b);
    ft_putstr_fd("ss\n", 1);
    return (OP_DONE);
}
```

Now: if either precondition fails, neither stack is touched. If both pass, both mutate, and the combined op is logged. All-or-nothing.

**Design layering note for future reference**: algorithm tiers (Phase 4-7) should be written so they never call `ss`/`rr`/`rrr` when one stack is ineligible — they have full visibility into both sizes and should plan accordingly. The atomic wrapper is a defense-in-depth safety net, not the primary mechanism. Two layers, each catching the other's mistakes.

**11 op すべてを手動トレース検証**

シーケンス:`sa → ra → rra → pb → pb → sb → ss → rb → rr → rrr → pa → ss(b size 1)`。全 wrapper が正しい stdout 行を出力、全状態遷移が期待通り、最終の `ss(b size 1)` で `"ss"` print が正しく抑制(no-op ケース)。

**設計問題を発見:`ss` / `rr` / `rrr` が silent に片方を mutate していた**

最終テスト(`ss` with b size 1)の出力を見て気付いた:**"ss\n" は print されていない(期待通り)が、stack a の top 2 要素は swap されている**。非 atomic 実装は `swap(a)`(成功)と `swap(b)`(NOT_DONE)を呼んでから print を抑制 —— でも `a` の mutation は残る。

**なぜ問題か**:`checker_42`(subject 必須の検証器)は **print された op 列を初期入力に適用し、program の最終状態と比較**する。print されていない op の結果が最終状態にあると → **KO**。dual-stack wrapper が片方 size 不足で呼ばれるたびに必ず検証失敗。

**修正:`ss` / `rr` / `rrr` を atomic 化**。両 stack の前提を **mutation 前に** check:

```c
t_operation  ss(t_stack *a, t_stack *b)
{
    if (a->size < 2 || b->size < 2)
        return (OP_NOT_DONE);
    swap(a);
    swap(b);
    ft_putstr_fd("ss\n", 1);
    return (OP_DONE);
}
```

これで:どちらかの前提未達なら両 stack を触らず、両方通れば両方 mutate + 複合 op を log。all-or-nothing。

**将来用の設計階層メモ**:アルゴリズム tier(Phase 4-7)は片方の stack が ineligible な時に `ss`/`rr`/`rrr` を呼ばないように書く —— 両 size の完全な可視性があるはずなので、計画段階で避けられる。atomic wrapper は **defense-in-depth の safety net** であって primary mechanism ではない。2 層が互いのミスを catch する構造。
