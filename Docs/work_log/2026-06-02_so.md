**Critical bug caught and discussed**

During T5 verification, my main.c initially had `t_stack *stack_a;` (still a pointer) but I was passing `&stack_a` to the new `stack_init(t_stack *)`. This is a type mismatch (`t_stack **` passed as `t_stack *`) that would silently corrupt adjacent stack memory. The compiler should warn, but I had skipped `make` and gone straight to running the binary. Lesson: always re-build after touching pointer-to-value type changes. The bug pair (declaration + struct access via `->` vs `.`) is exactly the kind of issue that compiles cleanly with both signatures but produces UB.

**Two design questions resolved during the session**

1. *Does each node need its fields reset before being freed?* — No. The free returns the memory to the allocator, after which any prior writes are irrelevant. Reset is only meaningful for memory we keep using (the t_stack struct itself — Javier's reset pattern applies there).
2. *Should `stack_free` be renamed to `stack_clear` (libft's convention for "free contents + reset state")?* — Considered, decided to keep `stack_free` for minimal churn. Both names defensible in review.
