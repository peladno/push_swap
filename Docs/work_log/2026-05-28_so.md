### What I did

**Phase 1 — M2 (stack primitives) completed**
- Implemented `stack_init()` and `stack_free()` on `so/phase1-stack-parser` branch.
- Started writing `stack_push_top()` but noticed an order issue during design: iterating argv left-to-right and pushing to top would put the **last** argv arg on top, contradicting subject VI.5 ("first argument is the top of the stack"). Two possible resolutions: iterate argv in reverse with `push_top`, OR push to bottom with normal iteration. Chose the latter and **renamed to `stack_push_bottom`**.
- Debugged several issues across `stack_free` and `stack_push_bottom`:
  - `stack_free`: NULL check ordering; loop variable shadowing of `next`; missing `free(s)` for the stack body itself.
  - `stack_push_bottom`: return-type mismatch (`int` declared but `return;` used), type mismatch on a temporary helper, missing updates to `top`/`tail`/`size`, and the function was effectively a push-to-tail under a misleading name.
  - Identified that the user-written `node_last(t_node *)` helper was effectively an O(n) walk that defeats the purpose of having `s->tail`. Removed and replaced with direct `s->tail` access.
- Built and tested:
  - `./push_swap 3 1 2` → first arg `3` is at top ✓
  - `./push_swap` (no args) → usage handled
  - valgrind: 5 allocs, 5 frees, **no leaks**.
- Decided to **not implement `stack_pop_top`** (YAGNI): no production path needs it (operations rewire existing nodes; parser only pushes; cleanup uses `stack_free`).
- Refactored into one-function-per-file layout for Norm-friendliness:
  - `src/stack/stack_init.c`
  - `src/stack/stack_free.c`
  - `src/stack/stack_push_bottom.c`
- Updated `implementation_plans/data-structure.md` to reflect the two M2 design changes (rename + removal) with rationale notes in the metadata block.
- **Committed M2** on the sub-branch.

**Learning — design discussions**
- Talked through *why* `stack_utils.c` exists and where its functions are (and are not) used. Established that `stack_push_bottom` is essentially a **parser-only** primitive; the 11 push_swap operations rewire pointers directly without calling these helpers.
- Walked through binary search (concept + why O(log n)) and confirmed that even Option B for `coord_compress` (sort + binary search) is O(n log n) overall — the sort cost is included but bounded.
- Compared candidates for the sort step of Option B (Bubble/Selection/Insertion are O(n²) and defeat the purpose; Quick Sort is the practical O(n log n) choice; `qsort` not allowed by subject). Concluded that **for Phase 1, the naive O(n²) Option A is the right choice** — implementation cost is low, the cost is invisible to subject's complexity model, and wall-clock is negligible at n=500.
